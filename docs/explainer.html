The core problem

A camera flattens the world

When a 3D scene is captured by a single camera, one entire dimension — depth — is collapsed. A small drill bit nearby and a tall lamppost far away can occupy exactly the same pixels. So how do we recover what was lost?

The information loss

A point at world coordinate (X, Y, Z) projects to a pixel at (u, v). The mapping divides by Z — the depth — which means an infinite number of 3D points map to the same 2D pixel. 

# Forward (camera projection)
u = f · X / Z + cx
v = f · Y / Z + cy

# Inverse — needs Z!
X = (u − cx) · Z / f
Y = (v − cy) · Z / f

Three ways to get Z back

  • Stereo cameras — two views, triangulate. Accurate but needs hardware.
  • LiDAR / depth sensors — measure directly. Best quality, expensive.
  • Monocular AI — predict depth from one image. Cheap, but relative not absolute.

YOLO-3D takes the third path — a deep network estimates depth, then we reconstruct geometry with classical math. The catch: the network only tells us how depths compare, not their absolute values.

Stage 01 · Frame intake

The simplest stage — but with one important detail: the pipeline keeps three copies of every frame. original_frame goes to the depth model. detection_frame gets the YOLO/Roboflow boxes drawn on it. result_frame is the final output canvas where masks and 3D wireframes are composited. Keeping them separate avoids feedback artefacts. 

ret, frame = cap.read()
original_frame = frame.copy()
detection_frame = frame.copy()
result_frame = frame.copy()

Stage 02 · Detect objects

Every detection becomes a tuple of [bbox, score, class_id, track_id]. With Roboflow, polygon mask points are also captured and stored in self._last_masks. The detect() method also draws on detection_frame for visual debugging. 

detection_frame, detections = detector.detect(detection_frame, track=True)
if hasattr(detector, 'draw_masks'):
    detector.draw_masks(result_frame)

Stage 03 · Estimate depth

Depth Anything v2 produces a normalised depth map (each pixel 0–1). The pipeline uses two strategies for sampling per detection: get_depth_at_point(cx, cy) for people and animals (centre point), get_depth_in_region(bbox, method='median') for rigid objects (more robust to noise around edges). 

depth_map = depth_estimator.estimate_depth(original_frame)

if class_name in ['person', 'cat', 'dog']:
    depth_value = depth_estimator.get_depth_at_point(depth_map, cx, cy)
else:
    depth_value = depth_estimator.get_depth_in_region(depth_map, bbox, method='median')

Stage 04 · Reconstruct 3D

The pinhole camera model converts pixel measurements into physical units. Two formulas drive everything: a linear depth-to-distance mapping, and the pinhole height equation. Both depend on configurable constants that you must match to your camera and scene. 

# Linear mapping
distance_m = depth_min + depth_value * (depth_max - depth_min)

# Pinhole inverse
height_m = (height_px / focal_length) * distance_m
height_cm = round(height_m * 100, 1)
The three models

Three brains, one decision

YOLO-3D is a hybrid system: two neural networks for perception, and a small geometric calculator for measurement.

Detection

YOLO26

"What's in the picture, and where?"

  • Real-time end-to-end detector — no NMS post-processing
  • Up to 43 % faster CPU inference than YOLO11
  • Outputs [bbox, class_id, confidence, track_id]
  • Five sizes: nano → extra (we use nano)
Depth

Depth Anything v2

"How far is each pixel?"

  • Monocular — needs only one camera image
  • Outputs a depth map in relative 0–1 range
  • Forced to CPU on Apple Silicon (MPS doesn't support all ops)
  • Per-class strategy: center-point for people, median for objects
Geometry

BBox3DEstimator

"Translate pixels and depth into metres."

  • Pinhole camera model
  • Per-object Kalman filter for temporal smoothing
  • Class-specific physical dimension priors (30+ classes)
  • Outputs height in cm, distance in m, 3D wireframe

Lab 02 · Depth mapping

The scale problem

Depth Anything outputs values in 0–1. Your scene exists in metres. Bridging the two is a choice with major consequences.

distance_m = depth_min + depth_value × (depth_max − depth_min)

↳ Why this matters: the depth model says "this pixel is roughly midway in the scene" but cannot tell you whether the scene spans 50 cm or 50 metres. You have to tell it.

The trap

Default values are someone else's decisions

YOLO-3D's defaults come from KITTI — a dataset of cars driving in Karlsruhe. Your phone's camera, looking at a tray of drill bits, is a radically different setup.

⚠ KITTI defaults on a phone close-up

Phone held ~50 cm from tray. Defaults assume objects are 1–10 m away with a 718 px focal length.

Object true height ~12 cm (drill bit)
Estimated by pipeline ~48 cm 🚨
Error factor ×4 too tall
Why? distance clamped to 1m; wrong f

✓ Calibrated for the actual scene

depth_min_m = 0.3, depth_max_m = 2.0, focal_length_px = 1100 (phone-typical).

Object true height ~12 cm (drill bit)
Estimated by pipeline ~14 cm ✓
Error factor +15 % off
Why? depth range matches scene

Code architecture

The module map

Click any module to see its responsibility, inputs, and outputs.

Click a module to see details

Each box represents a Python module in the project. The arrows show how data flows: detection results, depth maps, and configuration all converge in run.py, which orchestrates the pipeline.

run.py · the orchestrator

Loads config (depth range, focal length, API keys), instantiates the three models, and runs the per-frame loop. It owns the result_frame canvas that gets composited from detector boxes, Roboflow masks, depth visualisation, and the bird's-eye view panel. CSV export and FPS display also live here.

detection_model.py · YOLO26 wrapper

Wraps Ultralytics' YOLO("yolo26n"). Picks a device (CUDA → MPS → CPU), handles MPS-specific operation fallbacks, and returns detections in a uniform format [bbox, score, class_id, track_id]. Also draws boxes and tracking trajectories on the input frame.

roboflow_model.py · custom model + masks

Drop-in replacement for the YOLO detector. Calls the Roboflow hosted inference API, parses both bounding boxes and polygon mask points, and stores the latter in self._last_masks. The draw_masks(frame) method paints semi-transparent fills coloured by Dutch class name (blauw → blue, oranje → orange, etc.).

depth_model.py · Depth Anything v2

Wraps Hugging Face's depth estimation pipeline. Returns a normalised (H × W) depth map per frame. Provides two query helpers: get_depth_at_point(x, y) for centre-point sampling and get_depth_in_region(bbox, method) for box-region sampling (mean / median / min). Also produces a colourised heatmap for the inset.

bbox3d_utils.py · geometry + BEV

Two classes. BBox3DEstimator holds the camera intrinsic K matrix, the configurable depth_min / depth_max, per-object Kalman filters for smoothing, and class-specific physical dimension priors. It produces 3D box wireframes and the height-in-cm estimates. BirdEyeView renders the top-down panel.

.env · configuration secrets

Holds ROBOFLOW_API_KEY and ROBOFLOW_MODEL_ID. Loaded by python-dotenv at module import. Listed in .gitignore so it never reaches the repo. A tracked .env.example documents which variables are required.

Output · what reaches disk

Three output streams: an MP4 video file (output.mp4 by default), a CSV file with one row per detection per frame (frame, timestamp, class, id, conf, bbox, depth, distance, height_cm), and live OpenCV preview windows for monitoring. The CSV is what your downstream analysis or dashboards consume.

For your minor

Three lessons for data-driven decisions

YOLO-3D is a tiny system, but it teaches large lessons about what it means to make decisions from measured data.

01

Every measurement is a model

The "height" your pipeline reports is not measured — it is computed from pixels, an estimated depth, and a focal length. Change any of those and the answer changes. A number is only as honest as the assumptions behind it.

02

Defaults are decisions

The KITTI focal length and 1–10 m depth range were sensible choices for autonomous-vehicle data — but indefensible for a phone on a desk. Inheriting defaults silently inherits assumptions. Always ask: who picked these, and for what purpose?

03

Calibration is data work

The fix isn't smarter AI — it's better calibration. A measuring tape in the scene, a known reference object, or a one-time camera calibration step can turn a wildly wrong system into a reliable one. Ground-truth beats clever architecture.
